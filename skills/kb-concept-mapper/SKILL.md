# 知识库概念地图生成器（kb-concept-mapper）

## 功能定位

支持两种模式：
- **知识库模式**：用户指定一个知识库，自动扫描全部文档，提取核心概念并建立关联，生成概念地图
- **单文档模式**：用户提供一篇长文、一本书、一份报告等，直接从中提取核心概念和内部逻辑结构，生成概念地图

两种模式输出一致：一张**可交互、可拖拽、可展开**的概念地图（HTML 单文件）。

---

## 触发条件

用户说类似以下内容时触发：

**知识库模式：**
- "帮我扫描这个知识库"
- "把XX知识库画成概念地图"
- "生成概念图"
- "帮我看看这个知识库里都有什么概念"
- "做个知识图谱"

**单文档模式：**
- "把这篇文章画成概念地图"
- "给这本书做个概念图"
- "梳理这篇长文的核心概念"
- "这个概念图帮我分析下这个文档"
- "提取这篇文章的概念结构"

## 不适用场景

- 纯文档搜索/检索不生成图谱
- 知识库的元信息管理（用 ima-knowledge Skill）

## 完整执行流程

### ⚡ 模式判断（前置）

根据用户输入判断走哪条路径：

| 用户输入特征 | 模式 |
|------------|------|

| 模糊不清 | → 询问用户"是想扫描整个知识库，还是对某一篇具体文档做概念梳理？" |

### 模式一：知识库模式

### Step 1A-0：确定目标知识库

- 若用户已指定知识库名称 → 用 `get_knowledge_list` API 查找匹配，获取 `kb_id`
  - 若已知道 kb_id（如高频共享库）直接用
- 若用户未指定 → 询问用户想扫描哪个知识库
- 若 kb_id 解析失败 → 报错并返回

### Step 1A-1：全量扫描文档清单（关键改进）

**⚠️ 不要使用 `search(source="kb")` 来获取文档清单！**
- `search` 工具做的是相关性搜索，最多返回 15 条结果，且按匹配度排序
- 它会遗漏大量文档，尤其是文件夹深处的、标题不匹配搜索词的文档
- **正确做法：** 使用 `get_knowledge_list` API 递归遍历所有文件夹

#### 全量扫描算法

**输入：** `kb_id`（目标知识库 ID）

**输出：** 结构化的文档清单 `[{title, media_id, media_type, folder_path}, ...]`

**算法步骤：**

```
函数 scan_folder(kb_id, folder_id="", folder_path="根目录"):
    1. 调用 get_knowledge_list API：
       - endpoint: /openapi/wiki/v1/get_knowledge_list
       - limit: 50（最大值）
       - cursor: ""（首次为空）
       - knowledge_base_id: kb_id
       - folder_id: 当前文件夹 ID（根目录不传）
    2. 遍历返回的 knowledge_list：
       - media_type=99 → 子文件夹，递归调用 scan_folder()
       - 其他 media_type → 加入文档清单，记录 folder_path
    3. 如果 is_end=false，用 next_cursor 翻页继续
    4. 翻页最多 5 次（即单文件夹最多 250 条），避免无限循环
    5. 返回该文件夹下的所有文档 + 子文件夹文档

执行 scan_folder(kb_id) → 得到全量文档清单
```

**API 调用参考（来自 ima-knowledge Skill 的 reference 文件）：**

```bash
curl -s -X POST "https://ima.qq.com/openapi/wiki/v1/get_knowledge_list" \
    -H "ima-openapi-clientid: $IMA_OPENAPI_CLIENTID" \
    -H "ima-openapi-apikey: $IMA_OPENAPI_APIKEY" \
    -H "Content-Type: application/json" \
    -d '{"cursor":"","limit":50,"knowledge_base_id":"<kb_id>","folder_id":"<folder_id>"}'
```

**文档量阈值处理（基于真实全量总数）：**
- ≤ 10 篇 → 全量扫描 ✅
- 11~30 篇 → 提示用户"文档较多，可指定关注方向缩小范围"，若用户不缩小则全量
- > 30 篇 → **强制**让用户输入 2~3 个关注方向的关键词，AI 发散 3~5 个相关词后组合搜索，得到精准文档清单

**技术要点：**
- 文件夹 ID 带 `folder_` 前缀，从 `media_id` 字段获取
- `media_type` 判断：99=文件夹，非99=文档/笔记/文件（详见 media_type 速查表）
- 根目录遍历时 `folder_id` 参数不传（即 JSON 中不包含该字段）
- `content_count` 字段（在 search_knowledge_base 返回值中）可提前预估知识库体量

### Step 1A-2：批量读取文档内容

对清单中每篇文档，根据 media_type 选择读取方式：

| media_type | 类型 | 读取方式 |
|-----------|------|---------|
| 11（笔记） | 笔记 | `fetch(type="media_id", id=media_id)` |
| 7（Markdown） | 文本 | `fetch(type="media_id", id=media_id)` |
| 13（TXT） | 纯文本 | `fetch(type="media_id", id=media_id)` |
| 2（网页） | 网页链接 | `fetch(type="media_id", id=media_id)` 或直接访问 URL |
| 6（公众号文章） | 公众号 | `fetch(type="media_id", id=media_id)` |
| 1（PDF） | PDF 文件 | `export_media_for_ima_sandbox` 导出后下载，用 `fetch(type="file_path")` 读取 |
| 3（Word） | Word 文件 | `export_media_for_ima_sandbox` 导出后用 python-docx 或 `fetch(type="file_path")` 读取 |
| 9（图片） | 图片 | 图片无法提取文本概念，**跳过**但记录图片标题作为上下文线索 |
| 其他类型 | 其他 | 尝试用 fetch，失败则跳过并记录日志 |

**通用原则：**
- 短文档直接读全文
- 长文档用带 `question` 参数的方式获取核心内容（如 "请提取这篇文档的核心概念和方法论"）
- 所有可读文档内容合并为一个**文本语料库**（标注来源文档标题）
- 无法读取或空内容的文档跳过，记录日志
- 图片类文件不参与概念提取，但可辅助理解上下文

### Step 1A-3：概念提取 + 关联分析

这是核心 AI 环节。参考 `references/GUIDE_concept_extraction.md` 进行概念提取，输出的 JSON 数据结构如下：

```json
{
  "knowledge_base": "知识库名称",
  "total_docs": 15,
  "nodes": [
    {
      "id": "node_1",
      "label": "双脑循环",
      "weight": 0.85,
      "category": "方法论",
      "description": "将外部知识内化为个人认知模式的循环过程",
      "docs": ["doc_title_1", "doc_title_3"]
    }
  ],
  "edges": [
    {
      "source": "node_1",
      "target": "node_2",
      "relation": "包含",
      "strength": 0.9
    }
  ]
}
```

**概念提取原则：**
1. 只保留**有实质内容支撑**的概念，不凭空捏造
2. 同义词合并（如"AI""人工智能""大模型" → 按高频词合并）
3. 概念数量控制在 **20~80 个**之间（小于 20 说明文档太少，大于 80 地图变成毛线球）
4. 权重 `weight` 基于：文档出现频次 + 在文档中的重要程度，归一化到 0~1
5. 关系 `strength` 基于：共现次数/总文档数，归一化到 0~1
6. 概念分类依据：AI 从模板的 `CAT` 对象中选取最匹配的类别。若 `CAT` 已自定义为领域分类（如"心理阻碍""行动策略"），则使用自定义类别；若保持默认，则使用默认的 input/external/internal/output/practice/arch

### Step 1A-4：生成交互式概念地图（HTML）

**技术方案：纯 Canvas 引擎，零外部依赖。**

#### 使用 Canvas 引擎模板

用 `references/TEMPLATE_concept_map.html` 模板渲染概念地图。该模板内含一个完整的**自研力导向图引擎**（~300 行原生 JS）：

- **力模拟**：斥力（Coulomb）+ 引力（Hooke）+ 中心引力 + 碰撞检测 + 速度阻尼
- **Canvas 渲染**：背景网格 + 连线 + 标签 + 节点圆 + hover 光晕
- **交互**：节点拖拽、滚轮缩放（以光标为中心）、空白区域平移、hover 高亮关联、搜索过滤
- **触屏支持**：移动端拖拽和平移

#### 生成流程（⚠️ 必须逐条执行，不可跳过）

**核心方法：读取模板 → 用 file_edit 替换数据区域 → 更新标题 → 交付**

1. **复制模板**到目标位置：
   ```bash
   cp references/TEMPLATE_concept_map.html 概念地图_<知识库名称>.html
   ```

2. **替换节点数据**：用 `file_edit` 替换模板中标记的 `rawNodes` 区域
   - 定位锚点：`// ===== 【替换区域开始】将下方 rawNodes 替换为提取的概念数据 =====`
   - 替换范围：从该注释行到 `// ===== rawNodes 替换区域结束 =====` 的整段内容
   - 注意：引擎代码中有 `rawNodes.forEach(`（第 151 行），那是运行时遍历代码，**不要动它**
   - 数据格式（使用**单引号**，与模板一致）：
     ```javascript
     {id:'概念ID', label:'显示名称', cat:'分类', desc:'简要描述', docs:['来源文档1','来源文档2']}
     ```
   - cat 可选值：默认模板中定义了 input/external/internal/output/practice/arch 六类，
     ⚠️ 但**可根据文档内容自定义**——替换 `var CAT = {...}` 为领域更贴切的分类体系。
     例如做「周岭老师行动类」概念地图，可自定义为：
     ```javascript
     var CAT = {
       barrier:  {label:'心理阻碍',    color:'#FF6B6B'},
       engine:   {label:'内驱引擎',    color:'#4ECDC4'},
       tactic:   {label:'行动策略',    color:'#A29BFE'},
       advance:  {label:'学习进阶',    color:'#74B9FF'},
       result:   {label:'终极检验',    color:'#FFD700'}
     };
     ```
     自定义原则：分类数建议 **4~6 类**，每类用语义清晰的中文标签，颜色形成视觉区隔

3. **替换连线数据**：用 `file_edit` 替换模板中标记的 `rawEdges` 区域
   - 定位锚点：`// ===== 【替换区域开始】将下方 rawEdges 替换为提取的关联数据 =====`
   - 数据格式：`{s:'源节点id', t:'目标节点id', label:'关系标签'}`

4. **更新标题**：将 `<title>概念地图 — 知识库名称</title>` 改为对应的知识库名称

5. **检查完整性**：确保替换后的 HTML 文件结构完整，无残留的旧数据

**数据注入要点：**
- 所有字符串值必须用**单引号**（`'xxx'`），与模板格式一致
- JSON 中出现的**双引号**必须转义为 `\"`（因为数据写在 JS 字符串变量里）
- `docs` 字段是数组，即使只有一篇文档也写为 `['文档名']` 格式
- 确保替换后 `var rawNodes = [` 和结尾 `];` 完整无缺
- **`var CAT = {...}` 也要根据文档内容自定义**，替换锚点为 `// ===== CAT 替换区域结束（如需自定义请替换以上 CAT 对象）=====`
  - 如果默认分类（输入层/外化层/内化层/输出层/实践层）能准确表达内容，不改也行
  - 如果默认分类与文档内容不匹配（如周岭答疑的行动类），**必须**自定义，否则分类标签显得牵强
- 不要改动引擎代码，只替换数据部分

**数量控制：**
- 概念控制在 **20~80 个**，少于 20 说明文档太少，多于 80 地图变毛线球
- 每条边必须有明确的关系标签，不写空字符串

#### 输出文件特征

- **单文件 HTML，零外部依赖，完全离线可用**
- 打开即渲染，无加载等待
- 暗色主题（`#1a1b2e`），毛玻璃 UI 面板
- 文件体积 ~30KB（含全部数据和引擎代码）
- 包含所有交互功能：拖拽、缩放、hover、搜索、详情面板

### Step 1A-5：交付与调优

1. 用 `provide_file` 提供 HTML 下载链接
2. 告知用户文件已生成，并说明交互方式
3. 若用户提出修改意见（增删概念、调整关系、改样式等）：
   - 修改 JSON 数据 → 重新渲染 HTML
   - 若需大幅调整，从 Step 1A-3 重新执行

### 模式二：单文档模式

适用于一篇长文、一本书、一份报告等单一文档的概念地图生成。

#### Step 2B-0：定位并读取文档

根据文档来源选择读取方式：

| 文档来源 | 读取方式 |
|---------|---------|
| 知识库中的文档（media_id） | `fetch(type="media_id", id=media_id)` |
| 工作区文件（.md/.txt/.pdf/.docx） | `fetch(type="file_path", id=文件路径)` 或直接 file_read |
| 网页 URL | `fetch(type="url", id=URL)` |
| 用户直接粘贴的文本 | 直接使用文本内容 |

**读取策略：**
- 短文档（< 5000 字）→ 读全文
- 长文档（≥ 5000 字）→ 先用 `question` 参数获取核心内容（如"请提取本文档的核心概念、方法论和逻辑结构"），再根据需要补读关键章节
- 书籍类长文档 → 用 `question` 分别获取"整体框架"和"核心概念"，合并为语料库

**输出：** 一份文本语料库（标注来源文档标题）

#### Step 2B-1：提取概念

基于单文档的语料库提取概念。与知识库模式的区别：

| 维度 | 知识库模式 | 单文档模式 |
|------|-----------|-----------|
| 概念来源 | 跨文档提取 | 单文档内提取 |
| 概念数量 | 20~80 个 | 10~30 个 |
| 分类依据 | 跨文档主题聚类 | 文档内部逻辑结构（章/节/论点层级） |
| docs 字段 | 记录来源文档标题 | 记录来源章节/段落位置 |
| CAT 映射 | 按双脑循环层级 | 按文档内部逻辑角色映射 |

**单文档 CAT 映射原则：** AI 根据文档自身逻辑定义**最贴切的分类体系**，而非硬套默认分类。
以下仅为参考示例，每条概念选择最符合的分类即可：

| 概念类型 | 自定义分类示例 |
|---------|-------------|
| 背景/前提/问题/障碍 | 问题根源、心理阻碍 |
| 核心理论/模型/作者主张 | 核心理论、方法论 |
| 方法/步骤/策略/工具 | 行动策略、实践工具 |
| 外部引用/案例/数据 | 案例支撑 |
| 结论/方案/产出 | 终极检验、成果转化 |
| 人物/学派/思想源流 | 思想源头 |

**自定义 CAT 分类规范：**
- 分类数：**4~6 类**最佳
- 标签：中文，语义清晰
- 颜色：形成视觉区隔（避免相邻色系）
- 一致性：同类概念用同一分类

**概念提取原则（同知识库模式）：**
1. 只保留有实质内容支撑的概念，不凭空捏造
2. 同义词合并
3. 每条概念至少关联 1 个来源位置、1 条关系边，孤立节点不保留
4. 若文档内容浅薄（< 3 个实质概念），向用户说明效果有限

#### Step 2B-2：建立关联

分析概念之间的逻辑关系（因果关系、层级关系、对比关系、支撑关系、流程关系等），生成 edges 数据。

**单文档特有原则：**
- 关系应体现文档自身的论证逻辑，而非外部推测
- 优先建立"概念→子概念"的层级关系和"论点→论据"的支撑关系
- 关系标签应简洁明确（如"包含""支撑""-因果→""对比""前置""应用"）

#### Step 2B-3：生成概念地图 HTML

与知识库模式的 Step 1A-4 完全一致——使用同一套 Canvas 引擎模板，替换 rawNodes 和 rawEdges 数据。

**核心节点标记：** 单文档模式下，通常将文档核心命题/核心模型设为 r=32 的大节点。

**标题格式：** `<title>概念地图 — 文档名称</title>`

**标题栏格式：** `📄 文档名称 · 概念地图`

#### Step 2B-4：交付

与知识库模式 Step 1A-5 一致：provide_file → 告知交互方式 → 接受调优。

## 交互设计规范

概念地图 HTML 必须实现以下交互功能：

| 交互方式 | 效果 |
|---------|------|
| 🖱️ 拖拽节点 | 节点跟随鼠标，松开后力导向重新布局 |
| 🔍 滚轮缩放 | 整体缩放，始终保持中心点 |
| 🖱️ 悬停 | 显示节点名称 + 所属文档（tooltip） |
| 🔘 点击节点 | **展开** → 侧边面板显示概念详情 |
| 🔘 再次点击 或 关闭按钮 | **收起** → 回到概览 |
| 🔍 搜索框 | 输入概念名 → 高亮匹配节点 + 自动推近 |
| 🎨 颜色编码 | 按 category 区分颜色（legend 展示） |
| 📏 节点大小 | 按 weight 比例缩放 |
| 📐 连线粗细 | 按 strength 比例加粗 |

## 视觉风格规范

- 背景色：深色（`#1a1b2e`），Canvas 绘制淡网格线
- 节点：圆形 + 分类颜色填充 + 半透明白边，hover 时光晕高亮
- 连线颜色：与源节点分类颜色一致，hover 关联时变亮
- 标签：居中在节点圆内，长标签自动截断为 7 字符 + "…"
- 面板：毛玻璃效果（`rgba` + `backdrop-filter: blur`）
- 节点大小：核心概念（如 DBC）r=32px，次级概念 r=24px，普通概念 r=18px
- 所有 UI 元素：深色半透明底 + 细边框 + 圆角，与暗色背景融合

## 技术方案演进

### v1.0：纯 Canvas 自研引擎
- 零外部依赖，单文件自包含
- 自研 Verlet 积分力导向图模拟

- 模式判断前置：根据用户输入自动路由到知识库模式或单文档模式
- 单文档模式 CAT 映射：按文档内部逻辑角色（背景→论点→方法→结论）映射层级
- 概念数量分档：知识库模式 20~80 个，单文档模式 10~30 个

**关键教训：概念地图是独立交付给用户打开的文件，不能依赖任何外部资源。**

## 注意事项

1. 概念地图的**质量取决于文档内容**——内容浅薄或标题党会导致地图空洞
2. **完全离线可用**：无需网络，不加载任何外部资源，打开 HTML 即渲染
3. **知识库模式**：文档量 < 3 篇或内容不相关，向用户说明地图效果有限
4. **单文档模式**：实质概念 < 3 个时，向用户说明该文档不适合概念地图，建议改用深度阅读助手
5. **CAT 分类自定义**：默认 input/external/internal/output/practice 分类源自双脑循环框架。如果概念内容与默认分类不匹配（如周岭行动类），**必须**用 `file_edit` 替换 `var CAT = {...}` 为更贴切的领域分类，替换锚点为 `// ===== CAT 替换区域结束（如需自定义请替换以上 CAT 对象）=====`
6. **HTML 结构禁忌**：严禁 `<script><style>` 嵌套（曾致渲染崩溃），所有 CSS 放 `<head>` 的 `<style>` 内
7. **概念数量**：知识库模式 20~80 个，单文档模式 10~30 个
8. 交付后提醒用户：Ctrl+F 聚焦搜索、Esc 关闭面板、滚轮缩放、拖拽节点手动布局
